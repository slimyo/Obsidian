# 实验计划：工具增强 LLM Agent 在少样本时序场景的自适应策略

---

## 一、实验总体设计逻辑

五组实验形成递进的论证链，每组回答一个独立问题，合在一起构成完整的论文实验部分。

```
E1 主实验：少样本精度对比        ← 证明方法有效（能不能用）
    ↓
E2 置信度校准验证                ← 证明核心机制成立（置信度是否真实）
    ↓
E3 策略选择可解释性              ← 证明决策过程合理（Agent 是否在正确推理）
    ↓
E4 消融实验                      ← 证明各模块独立贡献（设计是否必要）
    ↓
E5 记忆机制泛化验证              ← 证明跨序列经验积累有效（记忆是否有用）
```

**核心假设（贯穿全部实验）：** 当训练数据量 N 极小（10~100 个时间步）时，对诊断结果的不确定性感知、保守策略选择和迭代反思机制，能够系统性地降低预测误差，且每个机制均有独立贡献。

## 二、数据集与少样本切割协议

### 2.1 数据集选择

|数据集|领域|总时间步|采样频率|选用理由|
|---|---|---|---|---|
|ETTh1 / ETTh2|电力负荷|17,420|1h|标准 Benchmark，审稿人熟悉|
|Weather|气象|52,696|10min|多变量，周期性复杂|
|ECL（电力消耗）|能源|26,304|1h|多变量，强季节性|
|M4（100条子集）|跨域混合|各异|各异|真实冷启动，异质性最强|
|ILI（流感样本）|医疗|966|周|本身就是短序列，天然少样本|

M4 取 100 条时选取 20 条月度 + 20 条季度 + 20 条年度 + 20 条周度 + 20 条日度，覆盖多种频率。ILI 因序列本身较短，直接用于验证极端少样本场景。

### 2.2 少样本切割协议（关键设计决策）

使用**绝对数量切割**而非比例切割（区别于 Time-LLM 的 5%/10% 方案）：

```
训练集：取序列前 N 个时间步
验证集：取训练集后 10 步（用于 Walk-Forward 验证和反思触发）
测试集：取训练集之后的 H 个时间步（H=96 为主实验）
```

五种数量级设置：

|设置|N（训练点数）|语义描述|对应真实场景|
|---|---|---|---|
|S1|10|极端冷启动|新安装传感器第 1 天|
|S2|20|严格少样本|新传感器第 1 周（日采样）|
|S3|50|中等少样本|新业务线约 2 个月数据|
|S4|100|相对充足|新业务线约 3 个月数据|
|S5|Full|全量参照|标准训练集（上限参照）|

**注意事项：** N=10 时，验证集无法留出 10 步，改为留出最后 3 步。当序列长度本身不足时（M4 年度序列），降级到 N=5 并在论文中单独说明。

## 三、基线方法

### 必选基线（5个，缺一不可）

**B1 - Naive（均值/漂移预测）** 纯统计无参数基线。均值预测 = 用训练集均值填充测试集；漂移预测 = 用最后两点线性外推。两者取较优结果。存在意义：任何有意义的方法都应超越它。

**B2 - ARIMA + ETS 自动选择** 用 Python `pmdarima` 的 `auto_arima` 和 `statsmodels` 的 ExponentialSmoothing 自动拟合，选 AIC 更低者作为统计基线代表。这是"无 LLM 介入时最强的传统方案"。

**B3 - LLMTime（Gruver et al., NeurIPS 2023）** 将时序序列化为文本数字串，直接调用 GPT-4o-mini 做 In-Context 预测，无任何工具调用或 Agent 规划。代表"有 LLM 但无 Agent 结构"的方案。

**B4 - TSci（Zhao et al., arXiv 2510.01538）** 直接在少样本设置下运行 TSci 原版，不做任何修改。这是最重要的基线，代表"有 Agent 但无不确定性感知"的当前最优方案。我们预期在 S1/S2 下 TSci 显著退化，这正是我们的贡献空间。

**B5 - Chronos-Small（Amazon, arXiv 2403.07815）** 使用预训练 Chronos 模型（60M 参数）做零样本预测，代表"有大规模预训练的基础模型"方案。

### 可选补充基线（有余力时加入）

**B6 - TimesFM**（Google ICML 2025，较新的预训练基础模型） **B7 - ForecastPFN**（NeurIPS 2023，隐式 RAG 方案） **B8 - N-BEATS / N-HiTS**（从头在少样本数据上训练的专用深度模型）

## 四、实验一：少样本精度主实验

### 4.1 实验目标

证明在不同程度的少样本场景（S1~S4）下，AdaptTS-Agent 在预测精度上系统性地优于所有基线，且随 N 增大优势逐渐收窄（这是符合预期的合理模式）。

### 4.2 实验设置

- **预测步长：** H ∈ {24, 48, 96}，主表用 H=96，附录给 H=24/48
- **数据集：** ETTh1、ETTh2、Weather、ECL、M4（100条）、ILI
- **重复次数：** 每组设置随机种子运行 5 次，报告均值 ± 标准差
- **LLM 骨干：** GPT-4o-mini（主实验），附录给 Qwen-Plus / Claude-3.5-Haiku 的跨模型鲁棒性验证

### 4.3 评估指标

**主指标：**

- MAE（平均绝对误差）：对量纲敏感，但直观
- MASE（平均绝对缩放误差）= MAE / 朴素预测MAE：跨数据集可比，消除量纲

**次要指标：**

- MSE（均方误差）：对极端误差更敏感
- SMAPE（对称平均绝对百分比误差）：M4 竞赛标准指标，便于与 M4 历史结果比较

**预期结果模式：**

```
N=10:  AdaptTS > TSci（差距最大，约 15-25%）
N=20:  AdaptTS > TSci（差距明显，约 10-18%）
N=50:  AdaptTS ≈ TSci or AdaptTS > TSci（差距缩小，约 5-10%）
N=100: AdaptTS ≈ TSci（差距趋近于 0）
Full:  AdaptTS ≈ TSci（全量时两者相当，我们不强求更优）
```

这个模式本身就是最有力的叙事：**我们的方法在 TSci 最脆弱的场景下发挥最大价值。**

### 4.4 结果表格格式（主表）

主表格式为"方法 × 数据集"，N 固定在 S2（N=20），因为 N=20 是"够用但不够多"的典型少样本场景，最能体现方法差异。额外提供"方法 × 数据量"的趋势图（折线图），横轴为 N，纵轴为 MAE，展示随数据量增加的性能收敛趋势。

## 五、实验二：置信度校准有效性验证

### 5.1 实验目标

**这是方法论核心的独立验证。** 我们的诊断层为每个结论附上高/中/低置信度标签，这个标签如果只是"装饰性的"而没有真实预测价值，整个方法的逻辑根基就不成立。本实验专门验证：置信度标签是否真正反映了诊断的可靠性，以及是否有效引导了策略选择。

### 5.2 实验设置

**步骤一：收集诊断-误差配对数据**

对所有实验序列，记录 Agent 在诊断阶段输出的三类标签：

- 趋势置信度（高/中/低）
- 周期性置信度（高/中/低）
- 平稳性置信度（高/中/低）

同时记录对应序列的最终预测误差（MAE）。以此构建"置信度-误差"配对数据集，共约 6 数据集 × 100条序列 × 5个N值 = 约 3,000 条配对记录。

**步骤二：校准误差计算**

借鉴 AUQ 框架（arXiv 2605.04295）的轨迹级 ECE（Expected Calibration Error）方法，按置信度分三个桶（高/中/低），计算每个桶内序列的实际平均误差，检验是否满足：

```
平均误差（低置信度桶）> 平均误差（中置信度桶）> 平均误差（高置信度桶）
```

定义**置信度-误差单调性比率（CMR）**：

```
CMR = (满足单调性的相邻桶对数) / (总相邻桶对数)
CMR = 1.0 → 置信度完美校准
CMR = 0.5 → 与随机相当
CMR < 0.5 → 置信度产生反效果（需警惕）
```

**步骤三：策略保守性验证**

统计低置信度场景下 Agent 选择集成策略（多策略组合）的比例，高置信度场景下选单一精细策略的比例，验证"置信度低→策略更保守"的设计逻辑是否真正被 LLM 执行。

**步骤四：对比两种置信度来源**

- 方案 A：统计检验的 p 值阈值（ADF p>0.1 → 平稳性低置信）
- 方案 B：LLM 语言表述中的主观置信度（LLM 说"可能有趋势"→ 中置信）
- 方案 C：双路交叉验证（A 和 B 均低置信才判为低置信）

比较三种方案的 CMR 值，确定最优的置信度来源策略。

### 5.3 核心结论预期

若 CMR ≥ 0.75，说明置信度标签有效，可作为方法的独立贡献之一报告。若 CMR < 0.6，需在 Discussion 中诚实讨论置信度标定的局限性，并说明保守策略选择本身仍有价值（即便置信度校准不完美）。

## 六、实验三：策略选择可解释性评估

### 6.1 实验目标

证明 Agent 的策略选择决策符合专业判断，而非随机或基于表面模式的决策。这是论文可解释性声明的定量支撑。

### 6.2 实验设置

**人工标注流程**

从实验数据集中精选 30 条典型序列，覆盖：

- 5 条强趋势 + 无明显季节性序列
- 5 条强季节性 + 平稳序列
- 5 条趋势 + 季节性叠加序列
- 5 条高噪声 + 无明显结构序列
- 5 条突变点序列（局部分布偏移）
- 5 条极端短序列（N=10 的真正冷启动）

将每条序列（只给前 N=20 个点）分别交给 3 位时序领域研究生标注"最合适的预测策略"（从策略库中选 1~3 个），3 人意见取多数投票作为"专家建议"。

**一致性指标**

计算 Agent 策略选择与专家建议的 Cohen's κ：

```
κ = (P_observed - P_chance) / (1 - P_chance)

κ > 0.61: 实质性一致（Substantial agreement）→ 可接受
κ > 0.80: 几乎完全一致（Almost perfect）→ 优秀
κ < 0.41: 中等以下一致性 → 需在论文中讨论
```

**额外分析：分场景一致性**

分别统计上述 6 类场景下的 κ 值，预期结果是：强结构序列（强趋势/强季节性）的 κ 较高，极端冷启动和高噪声序列的 κ 较低。这个模式本身就是一个有趣的发现，可以在论文中以图表呈现。

**与 TSci 的对比**

同样的 30 条序列，也统计 TSci 的策略选择与专家建议的一致性，与我们的方法对比。预期：我们的方法因为显式考虑了诊断不确定性，在低数据量下的决策更保守、更符合专家的谨慎判断。

### 6.3 标注者间一致性（重要细节）

在汇报 Cohen's κ 之前，先汇报三位标注者之间的 Fleiss's κ，若标注者间一致性 < 0.5，说明策略选择本身存在主观性，需在论文中说明这是任务固有难度而非方法缺陷。

## 七、实验四：消融实验

### 7.1 实验目标

量化四层设计中每一层的独立贡献，证明完整方法优于任何单独去掉某一层的变体。

### 7.2 消融变体设计

|变体编号|名称|描述|关闭的组件|
|---|---|---|---|
|Full|AdaptTS-Full|完整方法（四层全开）|—|
|A1|w/o Uncertainty|去掉置信度量化，改为点估计诊断|层一的置信度标签|
|A2|w/o Adaptive|固定策略选择规则（不随置信度变化）|层二的动态选择逻辑|
|A3|w/o Reflection|去掉迭代反思循环，只做一次预测|层三整体|
|A4|w/o Memory|去掉跨序列记忆，每次从空记忆开始|层四整体|
|A5|w/o UQ + Adaptive|同时去掉层一和层二（对应 TSci 的诊断方式）|层一+层二|
|A6|ICL-only|只用 LLM-ICL，不调用任何统计工具|工具调用整体|

### 7.3 实验设置

- **固定数据设置：** N=20（最能体现方法差异的设置），H=96
- **固定数据集：** ETTh1、Weather、ECL（三个足以说明问题）
- **重复次数：** 每个变体 3 次随机种子，取均值

### 7.4 预期结果与论证要点

预期贡献排序（MAE 降幅从大到小）：

```
层三（反思）> 层二（自适应选择）> 层一（置信度量化）> 层四（记忆）
```

原因：层三直接修正错误策略，效果最即时；层二是核心决策创新；层一是层二的基础；层四在单序列实验中贡献有限（需在 E5 中单独验证）。

若 A5 变体（去掉层一+层二）的结果接近 TSci，可进一步佐证：我们的改进主要来自不确定性感知设计，而非其他工程改进。

## 八、实验五：记忆机制与跨序列泛化

### 8.1 实验目标

记忆层（层四）在单序列实验中（E1~E4）贡献有限，这是正常的——记忆的价值体现在处理**一批相同领域的序列**时。本实验专门验证：随着处理序列数量增加，带记忆的系统是否能持续改善，而无记忆版本则无法做到。

### 8.2 实验设置

**场景构建：** 使用 M4 数据集中同一类别（如月度序列）的 50 条序列，按固定顺序逐条处理（模拟业务系统每天接入一条新序列的场景）。

**分组对比：**

- 组 A：AdaptTS-Full（带记忆）
- 组 B：AdaptTS w/o Memory（无记忆）

**记录指标：** 处理第 k 条序列时的预测 MAE，绘制 k=1,2,...,50 的 MAE 学习曲线。

**预期结果：**

```
组 A（带记忆）：MAE 随 k 增大整体呈下降趋势（偶有波动）
组 B（无记忆）：MAE 保持平稳，无系统性改善趋势
```

若组 A 的 MAE 在 k=20 之后稳定低于组 B，说明记忆机制有效积累了"有价值的经验"。

### 8.3 记忆负迁移检验（重要风险实验）

**场景设计：** 前 25 条序列来自强趋势类型，后 25 条来自强季节性类型（分布突变）。检验记忆中积累的趋势相关经验，是否会在遇到季节性序列时产生负迁移（即带记忆的系统在第 26~30 条序列上表现反而差于无记忆版本）。

若负迁移出现（组 A 在分布切换后短期 MAE 高于组 B），需在论文中诚实报告，并作为 Limitation 说明记忆的适用边界——这比隐藏负面结果更有学术价值。

### 8.4 记忆规模敏感性

测试记忆库容量 K（最多保存 K 条案例）对性能的影响，K ∈ {5, 10, 20, 50}。预期：K=10~20 为最优区间，K 过小记忆不足，K 过大引入噪声参考。

|实验|主指标|次要指标|统计检验|
|---|---|---|---|
|E1 主实验|MAE、MASE|MSE、SMAPE|Wilcoxon 符号秩检验|

## 十、实验实施细节

### 10.1 系统实现

|组件|技术选型|说明|
|---|---|---|
|Agent 编排框架|LangGraph|TSci 同款，便于对比复现|
|LLM 接口|OpenAI API (GPT-4o-mini)|主实验；可切换 Qwen-Plus 做对比|
|统计工具库|`statsmodels`, `pmdarima`, `scipy`|ADF/KPSS 检验、ARIMA、ACF/PACF|
|机器学习工具|`scikit-learn`, `xgboost`|XGBoost with lag features|
|时序基础工具|`sktime`, `darts`|Theta、ETS 等统一接口|
|记忆存储|`faiss` (CPU版)|向量检索，轻量无需 GPU|
|实验管理|MLflow|记录所有超参和结果，便于复现|

### 10.2 API 费用估算

```
E1: 6 数据集 × 5 N值 × 5 方法 × 5 次重复 = 750 次序列处理
    每次约 15 次 API 调用 × 平均 500 tokens = 7,500 tokens/次
    750 × 7,500 = 5,625,000 tokens
    GPT-4o-mini 费率 $0.15/M tokens → 约 $0.84

E2~E5: 约同量级，总计估算 ≈ $5~15
```

API 费用极低，风险可控。

### 10.3 单次实验运行时间估算

```
N=20, ETTh1, H=96, 单次：
  - Curator 诊断：2~3 次 LLM 调用 ≈ 15s
  - Planner 策略选择：2 次 LLM 调用 ≈ 10s
  - Forecaster 拟合（ARIMA + XGBoost + ICL）：≈ 30s
  - Reflection（若触发）：2 次 LLM 调用 ≈ 10s
  - 合计：约 60~90 秒/序列
```

E1 全量实验（750次）≈ 12.5~18.75 小时，可分多天并行运行。

### 10.4 随机性控制

- 所有 LLM 调用设置 `temperature=0.1`（不设为 0，保留少量随机性模拟真实使用）
- 统计模型拟合使用固定随机种子（1, 42, 123, 2024, 2025）
- 每组设置报告 5 个种子的均值 ± 标准差

**第 3 个月检查点的决策逻辑：** 若 Phase 1.5 的小实验显示 AdaptTS-Full 在 N=20 下比 TSci 的 MAE 改善 < 5%，则暂停全量实验，优先排查是哪个模块设计有问题。这个检查点是防止在错误方向上浪费时间的保险机制。

## 十二、风险预案

**风险 R1：置信度标定效果差（CMR < 0.6）** 预案：退回到"固定阈值保守策略"设计——当 N < 30 时无论诊断结果如何一律选集成策略，放弃自适应机制，改为规则驱动保守化。这是可以降级发表的版本。

**风险 R2：反思循环在噪声序列上发散** 预案：加入最大迭代次数硬限制（上限 3 次），并在触发上限时输出警告日志用于分析。把"发散案例"单独统计，作为方法局限性在论文中讨论。

**风险 R3：记忆机制产生大量负迁移** 预案：将记忆机制改为可选组件，主实验报告不含记忆的版本（AdaptTS w/o Memory），记忆机制作为"扩展分析"在附录呈现。保底方案仍然完整。

**风险 R4：GPT-4o-mini API 调用不稳定或费用超预算** 预案：切换到 Qwen-Plus（国内 API，更稳定）或本地部署 Qwen2.5-7B（16G 显卡可运行），主实验用 API，跨模型鲁棒性实验用本地模型。

**风险 R5：TSci 复现结果与原论文差异较大** 预案：直接使用 TSci 开源代码（GitHub Y-Research-SBU/TimeSeriesScientist）而非自行复现，确保基线数据的可靠性。若出现差异，以开源代码运行结果为准并在论文中注明。
