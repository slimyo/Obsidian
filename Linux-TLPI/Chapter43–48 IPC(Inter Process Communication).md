- 根据功能分为了三类：
	- 通信：关注进程间的数据交换
		1. 数据传输工具：区分这些工具的关键是写入和读取的概念，这些工具要求在用户内存和内核内存之间进行两次数据传输。进一步分为：
			1. 字节流：通过管道、FIFO及数据报socket交换的数据是一个无分隔符的字节流。
			2. 消息：以分隔符分隔的消息。一次读取一条完整消息
			3. 伪终端
		2. 共享内存：允许进程将数据放到进程间共享的内存完成信息交换。三种形式一般：
			1. System Ⅴ共享内存
			2. POSIX共享内存
			3. 内存映射
		- 数据传输和共享内存的差别：
			- 数据传输工具读取操作是破坏性的，尽管可以有多个读取者
			- 读取者和写者进程间同步是原子的。读取不包含数据的数据传输工具，默认阻塞直至写入了数据
			- 尽管共享内存通信速度块，速度上优势要弥补对共享内存上发生的操作进行同步不足的
			- 共享内存的数据对所有共享者进程可见（与破坏性读取不同）
	- 同步：关注进程和线程操作间的同步。UNIX系统提供下列同步工具：
		1. 信号量：一个信号量是一个由内核维护的整数，值不会小于0.进程可以增大或减小其值
		2. 文件锁：用来协调同一文件多个进程动作的一种同步方法。
		3. 互斥体和条件变量：mutex、cond（见chapter30）
	- 信号：尽管信号的主要作不在于次，但特定场景下仍然可将作为一种同步技术

### 管道和FIFO

- 概述：
	- 例`$ ls | wc -l`命令：
	 ![[tubing.png|500|500]]
	- 写入进程将其标准输出连到管道写入端，读取进程将标准输入连到管道读取端
	- 特性：
		- 一个管道是一个字节流：即不存在消息或消息边界概念。可按顺序读取任意大小数据块。不管写入进程和写入管道的大小
		- 从管道读取数据：从当前为空的管道中读取数据将被阻塞直到有一个字节被写入到管道为止
		- 管道是单向的：一端写入，另一端读取
		- 可以确保写入不超过PIPE_BUF字节操作是原子的：多个进程写入同一个管道，那么它们在一个时刻写入的数据量不超过PIPE_BUF字节，那么可确保写入数据不会发生混合的情况
		- 管道的容量是有限的：管道其实是一个在内核中维护的缓存器，存储能力有限。管道填满，后续写入将被阻塞直到移除了数据为止
- 创建和使用管道
	- `int pipe(int filedes[2])`
		- 成功的调用会在数组filedes中返回两个打开的文件描述符，一个读取端（0），一个写入端（1）。
		- 可以用read(),write().也能通过fdopen()获取描述符对应文件流后使用printf(),scanf()
		- 使用pipe()后调用fork()子进程可从继承的文件描述符副本获得文件描述符
	- 管道允许相关进程间通信：即进程间通过一个共同祖先进程创建管道即可
	- 关闭未使用管道文件描述符：不仅是确保进程不会耗尽其文件描述符限制.只有当所有进程所有引用一个管道的文件描述符被关闭后才会销毁该通道占用的资源
	- 与信号相比，管道可以用来协调一个进程的动作与多个其他相关进程匹配。信号不行（信号无法排队）
- 使用管道连接过滤器
- 通过管道与shell命令进行通信：`popen()`
	- 该函数创建一个管道，然后创建了一个子进程来执行shell，而shell又创建了一个子进程来执行command字符串
	- 与system()调用的区别：
- 管道和stdio缓冲：
	- stdio缓冲满与管道从一端发送数据到另一端的联系
- FIFO（有时称为命名管道）
	- 与管道类似，最大区别是其在文件系统中拥有一个名称，且其打开方式与打开普通文件是一样的。这样就能将FIFO用于非相关进程的通信。
	- 与管道一样，所有引用FIFO的描述符被关闭后，所有未被读取的数据被丢弃
	- `mkfifo`命令可以在shell中创建一个FIFO
	- 打开一个FIFO：一般，唯一明智的做法是在两端分别设置一个读取进程和一个写入进程。打开一个FIFO会同步读取进程和写入进程
- 使用FIFO和`tee(1)`创建双重管道线：
	- 使用FIFO能在管道线中创建子进程，这样，除了将一个进程的输出发送给管道线后一个进程之外，还可以复制进程的输出并将数据发送到另一个进程中。
- 使用管道实现一个客户端/服务器应用程序
- 非阻塞I/O：
	- 当一个进程打开FIFO一端时，如果另一端还没有打开，则该进程会阻塞。有时不期望阻塞，可以在调用open时指定O_NONBLOCK标记来实现非阻塞。同时对读取操作在没有数据可用时不阻塞或写时没有足够空间不阻塞也是有用的。

### System Ⅴ IPC

- 包括三种不同的进程间通信机制。
	1. 消息队列在进程间传递消息。与管道有点像，但：1.消息队列存在边界。2.每条消息包括一个整型的type字节
	2. 信号量允许多个进程同步它们的动作。
	3. 共享内存使得多个进程共享内存。

- System Ⅴ消息队列
	- 用来引用消息队列的句柄是一个由mssgget()调用返回的标识符。与文件描述符不同。
	- 通信面向消息，读者接受到写入的整条消息
	- 除了数据外，每条信息包含一个整数表示的类型
	- 消息队列无连接（不进行引用计数），程序难以知道何时能安全删除一个队列
- System Ⅴ信号量
	- def：一个由内核维护的整数，值不会小于0.进程可以增大或减小其值
	- 不是用于进程间传递数据的，用于同步进程的动作
	- 信号量本身没有意义，它的意义由使用信号量的进程赋予其的关联关系确定![[semopt.png|500|500]]
	- 常规步骤：
		- 使用semget()创建或打开信号量集
		- 使用semctl()操作初始化其中信号量
		- 使用semop()操作信号量值
		- 当所有进程都不需要使用信号量集后使用semctl()删除这个集合
	- 信号量的缺点：
		- 采用标识符而不是文件描述符引用
		- 创建和初始化都需要单独的系统调用，一些情况下要额外编程防止初始化竞争。
		- 内核不维护引用一个信号量集合的进程数量。难以确定何时删除和确保无用即删除。
		- 编程复杂。其他限制。
- System Ⅴ共享内存
	- 常规步骤：
		- 调用shmget()创建一个新共享内存段或取得一个共享内存段的标识符。
		- 使用shmat()来附上共享内存段，即使该段成为调用进程虚拟内存的一部分
		- 可以使用shmat()调用返回的addr值（进程虚拟内存地址中共享内存段的起点指针）引用该共享内存段
		- 调用shmdt()来分离共享内存段。（可选，进程终止自动完成）
		- 调用shmctl()来删除共享内存段。当前共享内存段的进程都分离后内存段才销毁。
