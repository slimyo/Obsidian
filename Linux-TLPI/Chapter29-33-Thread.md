- 概述：
	- 线程是允许应用程序并发执行多个任务的一种机制。同一程序中所有线程都会独立执行相同程序，且共享一份全局内存区域，其中包括初始化数据段、未初始化数据段、堆内存段。![[thread.png|500|400]]
	- 采用多进程的限制：
		- 进程间的信息难以共享。除了只读代码段外，父子进程未共享内存，必须采用进程间通信方式
		- 调用`fork()`来创建进程的代价相对较高。即便使用（copy-by-write）技术，仍然要复制诸如内存页表、文件描述符表等进程属性，开销不菲
	- 线程解决上述问题:
		- 线程间能方便快速共享信息：将数据复制到全局/堆（共享）变量中即可。（同步）
		- 线程创建速度快，无需复制内存页，页表等
	- POSIX线程API：Pthread API见附表
	- 线程和errno:
		- 传统unix api中errno为全局整型，然而多线程不行——引发竞争条件：线程调用函数通过全局errno返回错误时，会与其他发起函数调用检查errno的线程混淆
		- 每个线程均有属于自己的errno

- 创建线程
	- 启动程序时，产生的进程只有单条线程，称之为初始或主线程
	- `Pthread_create()`负责创建一条新线程
	```cpp
	#include<pthread.h>
	int pthread_create(pthread_t* thread,const pthread_attr_t *attr, 
						void*(*start)(void*),void *arg)
						//return 0成功，或者正数表示错误
	```
- 终止线程
	- 如下方式终止线程运行：
		1. 线程start函数执行return语句并返回指定值
		2. 线程调用`pthread_cancel()`
		3. 调用`pthread_cancel()`取消线程
		4. 任意线程调用了`exit()`，或主线程执行了return语句，都会导致进程中所有线程立刻终止
- 线程ID（Thread ID）
	- 进程内部每个线程都有一个唯一标识，称为线程ID。线程ID会返回给`pthread_create()`的调用者，一个线程可以通过`pthread_self()`来获取自己的线程ID
- 连接（joing）已终止线程
	- 函数`pthread_join()`等待由thread标识的线程终止。如果线程已经终止，其会立即返回
	- 若线程未分离，则必须使用`pthread_join()`来进行连接。如果未能连接，那么线程终止时将产生僵尸线程。浪费系统资源
	- 与进程中的`waitpid`区别:
		- 线程中对等关系
		- 。。。
- 线程的分离
	- 默认情况线程可连接的——线程退出时其他线程可以调用`pthread_join()`获取返回状态
	- 有时，不关心返回状态，只希望系统在线程终止时自动清理并移除它——可以通过调用`pthread_detach()`将该线程标记为分离状态
- 线程v.s.进程
	- 除了数据，多线程还可以共享某些其他信息
	-  除了前述进程的限制，线程也有缺点:
		1. 某个线程的bug可能危及该进程的所有线程，因为它们共享相同地址空间和其他属性
		2. 它们都在争用宿主进程中有限虚拟地址空间
		3. 多线程中处理信号要小心设计——一般避免使用
		4. 多线程应用，所有线程必须运行同一程序；多进程可以不同

# 线程同步

- 互斥量mutex(mutual exclusion缩写)：保护对共享变量的访问 
- `mutex`两种状态：
	1. locked：任何时候，至多只有一个线程可以锁定该互斥量。线程锁定互斥量即成为其所有者，只有所有者能给互斥量解锁（再次加锁可能阻塞或报错)
	2. unlocked
	3. 加锁`pthread_mutex_lock()`函数-->变体：`pthread_mutex_trylock()`、`pthread_mutex_timelock()`
	4. 解锁`pthread_mutex_unlock()`函数
- 死锁：
	- 发起`pthread_mutex_lock()`的线程自身之前已然将目标互斥量加锁
	- 常常发生在一个线程管理多个共享资源时
	- 解决办法：最简单定义互斥量层级关系，总是以相同顺序对多个互斥量进行锁定
- 静态分配互斥量：互斥量属于`pthread_mutex_t`类型变量，使用前要初始化
	- 静态初始化：`pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER;`
	- 只能对经由静态分配，且携带默认属性的互斥量进行初始化
- 动态初始化：
	- 如下情况必须使用`pthread_mutex_init()`
		1. 动态分配于堆的互斥量。如动态创建链表，表中每个结构包含pthread_mutex_t类型
		2. 互斥量是栈中分配的自动变量
		3. 初始化经由静态分配，且不使用默认属性的互斥量
	- 不使用了时要使用`pthread_mutex_destroy()`销毁
```cpp
# include<pthread.h>

int pthread_mutex_init(ptherad_mutex_t *mutex,const pthread_mutexattr_t *attr);
retruns 0 on success,or a positive error number on error
```
- 条件变量
	- 允许一个线程就某个共享变量的状态变换通知其他线程，并让其他线程阻塞于这一通知
	- 解决了等通知去执行操作问题，不用反复查询
- 通知和等待条件变量：
	- 发送信号：通知一个或多个处于等待状态的线程
		- `pthread_cond_signal`:只保证唤醒一条等待线程。（典型情况：各线程执行任务相同）
		- 或`pthread_cond_broadcast()`：唤醒所有线程。（典型情况：线程执行的任务不同）
	- 等待：收到通知前一直处于阻塞`pthread_cond_wait()`
		- 应该置于while循环中——`pthread_cond_wait()`返回时，不能对判断条件的状态做任何假设。（其他线程先醒、设计条件宽松、虚假唤醒等）
		- 条件变量和互斥量间天然的关联关系
- 静态分配的条件变量：
	- `phtread_cond_t cond = PTHREAD_COND_INITIALIZER;`
- 动态分配
	- `int pthread_cond_init()`
	-  `int pthread_cond_destroy()`
# 线程安全

- 线程安全函数：若函数和同时供多个线程安全调用，则称之为线程安全函数；反之则不能并发调用
- 实现线程安全函数的方式：
	1. 将函数与互斥量关联使用：函数中所有函数共享相同全局变量，应该将所有函数与该互斥量关联，调用函数时将其锁定，返回时解锁。（简单，但访问函数变成串行）
	2. 将共享变量与互斥量关联：确认函数中哪些部分是使用了共享变量的临界区，仅在临界区获取和释放互斥量
- 可重入函数和不可重入函数：
	- 可重入函数无需使用互斥量即可实现线程安全
	- 避免全局和静态变量使用
	- 需要返回调用者的任何信息，或是需要在对函数的历次调用间加以维护的信息，都存储于由调用者分配的缓冲区内
	- 非所有函数都可实现为可重入
- 一次性初始化
	- 满足：不管创建了多少线程，有些初始化动作只发生一次
	- 例子：库函数中需要初始化一个互斥锁，由于库函数的调用不可预知，（可能多个线程要调用），必须保证只初始化一次（锁的要求）
	- 库函数`pthread_onece()`
- 线程特有数据
	- 实现函数线程安全最有效的方法是使其可重入。不过对已有不可重入函数可能要修改函数接口。
	- 线程特有数据计算可无需修改函数接口，实现已有函数的线程安全。（效率偏低）
	- 使函数得以为每个调用线程分别维护一份变量的副本
- 线程局部存储
	- 类似上
	- 提供了持久的每线程存储

# 线程取消

- 函数`int pthread_cancel()`向指定线程发送取消请求
	- 立即返回，目标线程动作取决于线程取消状态和类型
- 线程取消状态及类型
	- `pthread_setcancelstate()`,`pthread_setcanceltype()`函数设定标志，允许线程对取消请求的响应过程加以控制
- 取消点：
	- 若将线程的取消性和类型分别设置为启用和延迟，仅当线程抵达某个取消点时，取消请求才会起作用。
- 线程可取消性的检查
	- 函数`pthread_testcancel()`产生一个取消点。线程如果已有处于挂起状态的取消请求，只要调用该函数，线程随之停止
	- 当线程执行代码未包含取消点时，可以周期性调用，确保对其他线程向其发送的取消请求做出及时响应
- 清理函数：
	- 设置一个或多个清理函数，线程遭到取消时自动执行这些函数
	- 线程执行到取消点取消时，要进行清理，防止出现共享变量、pthread变量的不一致问题，导致死锁等。
	- 增加清理函数`pthread_cleanup_push()`
	- 移除`pthread_cleanup_pop()`
- 异步取消：
	- 设置线程可异步取消时，可在任何时间点将其取消
	- 清理函数不清楚要执行哪些清理步骤
	- 一般原则：可异步取消的线程不应该分配任何资源，不能获取互斥量或锁

# 细节

- 线程栈：
	- 创建线程时，每个线程都有属于自己的线程栈，大小固定
- 线程和信号：
	- Unix信号模型基于进程模型设计，问世早于线程，二者有一些明显冲突，避免结合使用
	- 只要可能多线程应用设计应该避免使用信号
